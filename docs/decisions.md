## ADR-001 — 사진은 캡처의 필수가 아니다 (메모 단독 캡처 허용)

- **상태:** 확정 (2026-06-29)
- **맥락:** PRD 8-B는 사진을 필수로 두었으나, Mission은 "사진을 찍는다"가 아니라 "생각을 붙잡는다"이다. 책을 덮은 뒤·산책 중 등 찍을 페이지가 없을 때 사진을 강제하면 그 생각을 잃는다.
- **결정:**
  - 캡처는 **두 경로**를 가진다.
    - **사진 경로 (기본, 빠름):** 카메라 항상 켜짐 → 찰칵 → 태그 → Why → 저장. **3초 예산 적용.** 타이핑 없음.
    - **메모 경로 (사진 없음):** 텍스트 메모로 내용을 채운다. **타이핑 허용.** 3초 예산은 적용하지 않는다(의도적 느린 경로).
  - 즉 캡처의 "내용"은 *사진 또는 메모 중 최소 하나*가 있으면 유효하다.
- **함의:**
  - 데이터 모델: `Capture.image`가 nullable 가 된다. `image` 또는 `memo` 중 하나는 반드시 존재해야 한다(유효성 규칙).
  - 16장 Capture Loop Budget의 적용 대상은 "사진 경로"로 한정됨을 PRD에 명시해야 한다.
  - "캡처 루프를 느리게 하는 기능은 거부한다"(원칙 5)는 *사진 경로*에 대한 약속으로 재확인된다.

### ADR-001 보강 — "타이핑 금지"가 아니라 "타이핑 없이도 완료 가능"

- **맥락:** 음성이 안 되거나 손으로 칠 수밖에 없는 순간은 항상 존재한다. "타이핑 없음"은 과한 제약이었다.
- **결정:** 타이핑은 **모든 경로에서 항상 가능**하다(Why 자유입력, 메모). 단, 빠른 경로의 **완료에 타이핑이 필수는 아니다** — 칩 탭만으로도 끝낼 수 있다.
- **예산의 정의 정정:** 16장 Budget은 "타이핑 금지"가 아니라 **"타이핑 없이도 3초 안에 완료 *가능*하다"** 로 읽는다. 사용자가 원하면 언제든 친다.
- **함의:** PRD 10(Why)은 "칩 탭 + *항상 열려 있는* 선택적 자유입력"으로 기술한다. 자유입력은 숨겨진 기능이 아니라 늘 한 번에 닿는 곳에 있다.

## ADR-002 — 태그는 단일 선택

- **상태:** 확정 (2026-06-29)
- **맥락:** 태그(💡⭐🔗❓🌱)는 "AI에게 주는 대표 힌트". 복수 선택은 "다 골랐나?" 판단 + 완료 탭이라는 마찰을 더해 3초 예산을 깬다.
- **결정:** **한 캡처당 태그 1개.** 태그 탭 = 곧 다음 단계로의 진행(별도 확인 탭 없음). 뉘앙스(중요+의문 등)는 Why 칩이 두 번째 축으로 보완한다.
- **함의:**
  - 데이터 모델: `Capture.tag`는 단일 enum 값(nullable 여부는 ADR-004에서 결정).
  - 향후 복수 태그가 필요하면 별도 ADR로 재검토(되돌리기 쉬운 결정).

## ADR-003 — 사진은 클라이언트 리사이즈+압축 후 Blob으로 저장, 원본 미보관

- **상태:** 확정 (2026-06-29)
- **맥락:** 원본 사진(4–12MB)을 그대로 쓰면 저장이 느려 "카메라 즉시 복귀"가 깨지고, 수백 장에서 IndexedDB 쿼터에 부딪힌다. 반대로 과하게 줄이면 책 문장이 안 읽혀 캡처가 무의미해진다.
- **결정:**
  - 저장 직전 클라이언트에서 **리사이즈+압축**: 긴 변 ~1600px, JPEG 품질 ~0.7 → 목표 **200–500KB/장**. (책 본문 글자 가독성 유지 / 추후 튜닝 가능한 파라미터)
  - IndexedDB에 **Blob 자체로** 저장(Base64 금지 — 33% 비대 + 인코딩 비용).
  - **원본 미보관.** 자산은 "생각"이지 고해상도 사진이 아니다(원칙 3). *되돌리기 쉬운 결정* — 후속 OCR/정밀 확대 요구가 생기면 재검토.
  - 압축은 **저장 직후 백그라운드**로 수행하여 카메라 복귀를 막지 않는다.
- **함의:**
  - 리사이즈 파이프라인(canvas/`createImageBitmap` → `toBlob`)이 캡처 루프 성능의 핵심 → 카메라 웜업과 함께 최우선 프로토타입 대상.
  - 용량 모니터링: `navigator.storage.estimate()` 로 쿼터 추적, 임계 시 사용자에게 Export/정리 유도(별도 결정 사안).

## ADR-004 — 캡처 유효성 규칙: (사진 또는 메모) + 태그 필수, Why는 선택

- **상태:** 확정 (2026-06-29)
- **맥락:** 사진·메모·Why가 모두 선택이면 빈 캡처가 쌓이고, 모두 필수면 빠른 경로가 느려진다. 최소 조건을 정의해야 한다.
- **결정:**
  - **콘텐츠:** `image` 또는 `memo` 중 **최소 하나** 필수. (둘 다 없으면 캡처로 인정하지 않음)
  - **태그:** **필수**(단일). 한 번 탭 = 진행이라 비용 ~0, AI 힌트/Review 그룹핑의 핵심.
  - **Why:** **선택.** 좋은 문장을 본 순간 "왜"가 늘 떠오르진 않는다 → 강제 시 마찰. 대신 Review에서 "Why 없는 캡처"를 모아 사후 보강하도록 유도.
- **PRD 변경점(중요):** PRD 8-B·10은 Why를 *필수*로 두었으나 본 ADR에서 **선택으로 완화**한다. PRD 갱신 필요.
- **함의(데이터 모델 유효성):**
  - `image: Blob | null`, `memo: string | null` → `NOT (image IS NULL AND memo IS NULL)`.
  - `tag: TagEnum` (NOT NULL).
  - `why: string | null`.

## ADR-005 — 세션 생명주기: 명시적 시작 + 느슨한 자동 종료, 책 단위 이어읽기

- **상태:** 확정 (2026-06-29)
- **맥락:** 사용자는 "독서 끝"을 거의 누르지 않는다. 세션이 안 닫히면 다음 책 캡처가 엉뚱한 세션에 붙는다. 또한 한 책을 여러 날에 걸쳐 읽는다.
- **결정:**
  - **Book ↔ Session = 1:N.** 한 책은 여러 세션을 가진다. 책 재등록 없음(같은 Book UUID 재사용).
  - **시작(명시적):** 책 선택으로 세션 시작.
  - **종료(느슨, 3경로):**
    1. 사용자가 "세션 종료" 명시적 누름.
    2. **비활동 자동 종료:** 마지막 캡처 후 **8시간** 경과 시 자동 `Ended`(= 마지막 캡처 시각으로 채움). *튜닝 가능한 파라미터.*
    3. 다른 책으로 새 세션 시작 시 이전 열린 세션 자동 종료.
  - **이어읽기(2종, 둘 다 지원):**
    - *세션 이어가기:* 아직 `Ended` 안 된 세션 → 홈에서 "이어 읽기"로 같은 세션에 계속 캡처.
    - *책 다시 펴기:* 지난 세션이 닫혔으면 같은 Book에 묶인 **새 세션** 시작.
  - **중단 복구:** 세션·캡처는 매 단계 IndexedDB에 즉시 커밋 → 앱 강제 종료 후 재실행 시 "○○ 세션 이어갈까요?"로 복구.
- **함의:**
  - Review/Export 범위 선택: **"이번 세션" vs "이 책 전체"** 둘 다 제공.
  - `Session.ended`는 nullable(열린 세션 = null). 자동 종료는 앱 진입 시 lazy 평가로 처리(백그라운드 타이머 불필요).

## ADR-006 — 책 등록은 MVP에서 수동 입력만, ISBN 바코드 스캔은 Phase 2+

- **상태:** 확정 (2026-06-29)
- **맥락:** 책 등록은 세션 시작 단계(3초 예산 밖)지만, 귀찮으면 사용자가 대충 등록해 데이터가 더러워진다. 종이책엔 ISBN 바코드가 있다.
- **결정:**
  - **MVP: 수동 입력만.** 제목(필수), 저자(선택), 표지 사진(선택). 그게 전부.
  - **ISBN 바코드 스캔 + 메타데이터 조회는 Phase 2+로 연기.** 바코드 디코딩(`BarcodeDetector`/zxing) + ISBN→메타 외부 API는 MVP "서버/외부 API 없음"(PRD 19) 원칙과 충돌하고 복잡도만 키운다.
  - 근거: 책 등록은 *책당 1회*라 빈도가 매우 낮음 → MVP 단계 기술 투자 가치 낮음.
- **함의:** `Book` 필수 필드는 `title`만. `author`·`isbn`·`cover`는 nullable.

## ADR-007 — MVP Review는 "분석"이 아니라 "규칙 기반 분류 + 프롬프트 빌더"

- **상태:** 확정 (2026-06-29)
- **맥락:** PRD 8-C의 "주제/반복/관계"는 의미 분석이라 AI 없이는 불가. MVP는 AI를 부르지 않는다. Review가 *실제로 계산 가능한 것*만 정직하게 정의해야 한다.
- **결정:** MVP Review는 AI인 척하지 않는다. 규칙으로 묶을 수 있는 것만:
  - **태그별 묶음/카운트** (⭐ 중요 N, ❓ 의문 N, 🌱 아이디어 N …)
  - **Why 칩별 묶음** ("글감" 후보 N, "프로젝트에 써먹기" N) — PRD의 "글감 후보/인터뷰 추천"을 규칙으로 근사.
  - **시간순 타임라인** + **Why 없는 캡처** 별도 수집(사후 보강 유도, ADR-004).
  - "주제/반복/관계" 의미 분석은 **하지 않는다** → 프롬프트로 만들어 외부 LLM에 위임.
- **Review의 결론:** 화면 안 인사이트가 아니라 **"이 구조를 Export"** 버튼. 진짜 분석은 Export→외부 AI가 수행.

## ADR-008 — Export는 "prompt.md + 사진"을 파일로 함께 공유(Web Share API 1순위)

- **상태:** 확정 (2026-06-29)
- **맥락:** "프롬프트만 클립보드 복사"로는 부족하다. 외부 LLM이 OCR을 돌리려면 **사진이 첨부파일로** 전달돼야 하는데, 클립보드는 이미지를 첨부처럼 싣지 못한다(텍스트 용량은 문제 아님). `navigator.share({text, files})`의 `text:`는 받는 앱이 버릴 수 있다.
- **결정:**
  - **모든 콘텐츠를 파일로 공유.** `text:` 필드에 의존하지 않는다.
    ```
    navigator.share({ files: [ prompt.md, capture-01.jpg, capture-02.jpg, ... ] })
    ```
  - **`prompt.md` = 자기완결형 한 파일:** 지시문 + **태그 범례**(💡=흥미 …) + **구조화된 캡처 데이터**(태그·Why·메모·시각 + ADR-007 규칙 분류 결과) + **OCR 지시**("첨부 사진을 OCR해 각 캡처를 번호/시각으로 엮어라").
  - **요청 작업(고정, Review가 못 한 것):** 주제 추출 → 반복되는 생각 → 캡처 간 관계 → 글감 후보 → 인터뷰 질문 → 독서노트 초안.
  - **일관성 = 버전된 고정 템플릿**(`prompt-template v1`). 책·세션·LLM이 달라도 같은 행동.
  - **전달 경로 (우선순위):**
    1. **1순위(모바일):** Web Share API로 `prompt.md` + 사진 파일들 공유.
    2. **보조:** **"프롬프트 복사"**(클립보드, 웹 UI 붙여넣기가 가장 확실) + **ZIP/이미지 다운로드**(다중 이미지 업로드).
  - **범위:** "이번 세션" 또는 "이 책 전체"(ADR-005).
- **리스크(프로토타입 검증 목록 — 카메라 웜업과 함께):**
  - ChatGPT/Claude 앱이 **`.md` + 다중 이미지 혼합 공유**를 한 메시지로 받아주는지 실측 필요.
  - LLM이 `prompt.md`를 자동으로 "지시"로 인식하지 않을 수 있음 → 사용자가 "이 파일대로" 한 줄 입력 보완. 그래서 "프롬프트 복사" 보조 유지.
- **Phase 2 씨앗:** 세션이 사진 多이면 LLM이 버거움 → **"이 캡처만 내보내기"(선택적 Export/OCR)** 가 자연히 유용해짐.
- **미결:** 프롬프트 템플릿을 **고정 1종 vs 목적별 다종**(독서노트/블로그글감/인터뷰)으로 시작할지 — ADR-009.

## ADR-009 — 프롬프트 템플릿: MVP 고정 1종, Phase 2에서 Export 시점 선택

- **상태:** 확정 (2026-06-29)
- **맥락:** 같은 캡처라도 "독서노트/블로그 글감/인터뷰 질문"은 AI 지시가 다르다. 다종은 결정 마찰 + 유지보수 비용.
- **결정:**
  - **MVP: 고정 1종.** 한 응답에 독서노트 초안 + 글감 후보 + 인터뷰 질문을 *모두* 담게 한다(ADR-008의 요청 작업).
  - **Phase 2: Export 시점에 목적 선택**("독서노트만" / "블로그 글감" / "인터뷰") — 실제 요구가 보이면 추가. 되돌리기 쉬운 결정.

## ADR-010 — 디자인 언어: 토스(Toss)-derived 밝고 깨끗한 스타일, 카메라만 다크

- **상태:** 확정 (2026-06-29)
- **맥락:** 어두운/질감 있는 세 방향(암실·필드노트·뷰파인더) 제안 후, 사용자가 "토스 같은 느낌"을 선택. 토스의 본질은 어두움이 아니라 *군더더기 없는 자신감 + 친근함*.
- **결정:** 상세는 [`docs/design-language.md`](design-language.md). 요지:
  - 밝은 토스-클린이 앱 기본. **카메라 뷰파인더만 유일한 다크 풀블리드**(라이브 영상이 곧 색).
  - 무채(잉크 `#191F28`/그레이/화이트) + **파랑 하나 `#3182F6`**. 컬러는 카메라와 태그에서만 터짐.
  - **Pretendard 단일 패밀리**로 위계(토스식 절제). 큰 볼드 헤드라인 + tabular-nums.
  - Radius 둥글게, 하단 풀폭 파랑 CTA, 스프링 모션(단 캡처 루프 <100ms — 16장 예산).
  - **Signature = "저장의 쾌감"** — 셔터→태그 팝→만족스러운 저장 확인→즉시 복귀.

## ADR-011 — 캡처 예산 재정의: "앱 지연"만 측정, 사람 판단 시간 제외 (실기기 검증)

- **상태:** 확정 (2026-06-30)
- **맥락(실기기 측정):** GitHub Pages 배포본을 실제 폰에서 측정.
  - **웜업:** 첫 실행은 큼(권한 팝업+콜드 스타트, 1회성), **2회차 738ms** — 1초 예산 통과. ⇒ **최대 리스크(카메라 웜업 ≤1s)가 실기기에서 깨짐.** PRD 전제 성립.
  - **`capture`(셔터→저장):** 사용자 입력 속도에 좌우(막 누르면 ~1910ms, 천천히 더 큼). 즉 *앱 성능이 아니라 사람 판단 시간*을 재고 있었다. 앱 반응(셔터→동결, 탭→시트, 저장→복귀)은 체감상 빠름.
- **결정:**
  - PRD §16 "캡처 ≤3s"의 본뜻은 **"앱이 끼어드는 시간이 짧다"**이지 "사람이 3초 안에 생각하라"가 아니다. 측정을 **둘로 분리**:
    - **`appMs`(예산 대상):** 앱이 끼어든 시간의 합 = 셔터 동결 + 저장 커밋·복귀. **목표 ≤ 300ms.**
    - **`humanMs`(참고용):** 사람 판단 시간. 합격/불합격 아님. HUD에 표시만.
  - **`warmupMs` ≤ 1000ms** 예산은 유지(실기기 통과). 첫(콜드)/이후(웜)을 구분해 본다.
- **함의:** `BUDGET.captureMs` 폐기 → `BUDGET.appMs=300` 도입. HUD는 `warmup / app / 사람 / compress / img`로 표시. PRD §16 갱신.

## ADR-013: iOS Safari 호환 — 이미지 로드는 `Image`+`onload` (createImageBitmap·decode() 금지)

- **상태:** 확정 (2026-06-30)
- **맥락(실기기 버그, 2회 반복):** Export 단일 PDF 기능 배포 후, **iPhone Safari에서 "PDF 받기" → 실패**. 헤드리스 chromium(Playwright)에서 `buildPdf`를 재현하면 **정상**. 즉 Blink는 통과, **WebKit(iOS)에서만 throw**. 이 개발 환경에선 **WebKit 직접 실행 불가**(libgtk-4 등 시스템 라이브러리 누락)라 로컬 재현 불가 → 실패 토스트에 **실제 에러를 노출**해 온디바이스로 원인을 받아냈다.
  - **1차 원인:** `createImageBitmap(blob)` — iOS Safari 15~16에서 미지원/버그로 throw.
  - 1차 수정으로 `Image`+`decode()`로 바꿨더니 → **2차 에러 `EncodingError: Loading error.`** (실기기 토스트로 확인). iOS Safari는 **blob URL 이미지에 대해 `img.decode()`가 거부하는 WebKit 버그**가 있다(실제로는 `onload`로 정상 로드됨 — 앱의 상세/리뷰 화면이 같은 blob을 object URL로 이미 잘 표시).
- **결정:**
  - 이미지 로드는 `createImageBitmap`도 `decode()`도 쓰지 않고 **`Image` + `onload`/`onerror`** 만 쓴다(가장 호환성 높음). (`src/lib/pdf.ts`의 `loadImage`)
  - PDF 등 사용자 트리거 실패는 **실제 에러 메시지를 토스트에 노출**해 온디바이스 자가 진단이 되게 한다.
  - **iOS 전용 동작은 실기기로 검증**한다. 헤드리스 chromium 스모크(`npm run test:pdf`)는 총체적 깨짐을 잡는 **회귀 가드일 뿐 iOS 보증이 아니다**(이번 버그도 chromium은 통과했음).
- **함의(교훈):**
  - 이 앱은 **iOS 설치형 PWA가 주 타깃** → 항상 **WebKit↔Blink 차이를 의심**한다. 브라우저 API를 쓰기 전 WebKit 지원/버그 이력을 확인.
  - WebKit 취약 API 주의 목록: `createImageBitmap`(이번 건), `OffscreenCanvas`(이미 `image.ts`는 일반 canvas 폴백 있음 — 유지), Web Share의 파일 지원 등.
  - "한쪽 엔진에서 됐다"는 다른 엔진 보증이 아니다. 모바일 핵심 경로는 가능하면 실기기/실엔진(WebKit)로 확인.

## ADR-014 — 캡처 모델 개편: passage/note 분리, why 칩 제거, image‖passage 유효성, 사진/입력 모드 토글

- **상태:** 확정 (2026-06-30)
- **맥락:** 초기 모델(ADR-001/004)은 콘텐츠를 "사진 또는 메모"로 이분했고, Why 칩(`WHY_CHIPS`)이 "저장 의도"를 담당했다. 실제 사용 패턴을 보니 두 가지 문제가 드러났다.
  1. **책의 글(인용)과 내 생각이 `memo` 한 필드에 섞인다.** 독서노트에서 "저자가 쓴 것" vs "내가 쓴 것"은 구분이 핵심이다.
  2. **Why 칩이 `memo`(자유 텍스트)와 역할이 겹친다.** 칩 목록이 굳어진 코드 상수(`WHY_CHIPS`)로 남고, 실제 사용에서는 메모 자유 입력으로 흡수되는 경향.
- **결정:**
  1. **`passage` 필드 추가** (`string | null`) — 책에서 담고 싶은 글/인용. "저자의 문장". `image`와 함께 *책 내용*의 두 입력 형태.
  2. **`memo` = `note` 역할로 재정의** — 내 생각·주석. Why(저장 의도)를 흡수한다. 필드명은 코드 호환성을 위해 `memo` 유지.
  3. **유효성 규칙 갱신:** `(image IS NOT NULL) OR (passage가 있음) OR (memo가 있음)` + `tag`. 세 경로 모두 단독 유효. 기존 `image OR memo` 규칙의 자연스러운 확장.
  4. **WHY_CHIPS 상수 제거.** `src/db/types.ts`에서 삭제(모든 import 소거 후). `why?` 필드는 **레거시 읽기 전용으로 보존** — 기존 DB 레코드 표시/Export 시 `note`로 합쳐 표시.
  5. **캡처 화면 2-모드 토글:** "사진 모드"(카메라+셔터, 기존 빠른 경로) / "입력 모드"(passage+note 텍스트 입력). 세션 시작 시 모드를 선택하고, 캡처 화면 안에서 언제든 토글 가능.
  6. **세션 시작 선택 화면:** 책 선택 후 "사진 위주 독서" vs "텍스트 위주 독서" 중 초기 모드를 선택.
- **ADR 갱신 메모:**
  - **ADR-007 갱신:** Review의 "Why 칩별 묶음" 분류는 `why` 필드 의존에서 `memo`(note) 기반으로 전환. Why 칩 고정 목록 제거로 칩별 집계는 삭제; 태그별 묶음 + 텍스트 note는 그대로.
  - **ADR-009 갱신:** Export `prompt.md` 템플릿에서 Why 칩 항목 제거; `passage`(인용)과 `note`(내 생각)를 각각 구분 출력. 레거시 `why` 값은 `note`로 합쳐 표시.
- **함의:**
  - `isValidCapture`가 `passage`를 세 번째 콘텐츠 조건으로 받는다.
  - UI: 사진 모드에서 note 입력 가능(why 대체); 입력 모드에서 passage + note 각각 입력.
  - 레거시 호환: `Capture.why?`를 읽는 코드(Review, Export, 상세 화면)는 `why`를 `note`(memo)로 병합해 표시하면 기존 데이터가 그대로 보인다.

## ADR-015 — 이미지 IDB 저장은 ArrayBuffer(Blob 금지) — iOS IDB-Blob 버그 회피

- **상태:** 확정 (2026-06-30)
- **맥락:** iPhone 실기기 진단으로 근본 원인 확정: iOS Safari는 IndexedDB에 저장한 Blob을 나중에 읽을 때 `NotFoundError: The object can not be found here.`를 던진다(레코드는 남되 바이트가 사라지는 알려진 WebKit 버그). Android(Blink)는 정상. 이것이 Export PDF에 사진이 안 들어가던 진짜 원인이었다. ADR-003의 "Blob으로 저장" 방침이 iOS에서 깨진다.
- **결정:**
  - IndexedDB에 이미지를 **ArrayBuffer로 저장**(Blob 직접 저장 금지). iOS는 ArrayBuffer를 안정적으로 보관한다.
  - 변환은 **`src/db/db.ts` 경계에 격리**: 저장 시 `toStored(Capture) → { ...c, image: ArrayBuffer, imageType: string }`, 읽기 시 `fromStored(rec) → Capture(image: Blob)`.
  - 소비자(screens, lib/) 코드는 계속 `Capture.image: Blob | null`을 받아 **변경 없음**.
  - 옛 Blob 레코드(Android 기존 데이터)는 `fromStored`가 그대로 통과시켜 **양방향 호환** 유지.
- **ADR-003 개정 메모:** "Blob으로 저장" → "ArrayBuffer로 저장, 읽기 시 Blob 재구성"으로 개정.
- **함의:**
  - `addCapture`/`updateCapture`: `await toStored(c)` 후 put. 반환값은 원래 `Capture`(소비자 불변).
  - `getCapture`: `fromStored` 적용.
  - `capturesForSession`/`allCaptures`: 결과 배열에 `.map(fromStored)` 적용.
  - iOS 기존 사진: 이미 못 읽으므로 복구 불가 — 신규 캡처부터 정상.
  - 용량 변화 없음(base64 33% 비대 없이 ArrayBuffer 직접 저장).

## ADR-016 — 세션=회독 재정의: 책 중심 홈, 자동 종료 폐지, 회독 번호 계산

- **상태:** 확정 (2026-07-02)
- **맥락:** ADR-005의 세션 생명주기(느슨한 자동 종료 8h + 타책 시작 시 종료)는 "독서 세션"을 기본 단위로 두었지만, 실제 사용 단위는 **책**이다. 자동 종료는 사용자가 의도하지 않은 세션 분절을 만들었고(하루 지나 캡처하면 새 세션), 홈의 "이어 읽기" 판단을 복잡하게 했다. 사용자 멘탈 모델은 "이 책 N번째 읽기(회독)"에 가깝다.
- **결정:**
  1. **세션 = 회독(re-read)으로 재정의.** Session 레코드는 유지하되 의미를 "한 책의 N번째 읽기"로 바꾼다. 데이터 마이그레이션 없음.
  2. **회독 번호는 저장하지 않고 계산** — 그 책의 세션을 `started` asc 정렬(JS sort 필수, `roundNumberOf`) 후 1-based 순번. (번호 수동 수정은 `Session.roundNo` 선택 필드로 후속 태스크에서.)
  3. **`session.project` = 회독 제목.** "왜 이 책을 읽나요" 목적 텍스트를 회독 제목으로 재해석. Review에서 **사후 편집** 가능.
  4. **자동 종료 삭제:** 8h 비활동 자동 종료, "다른 책 시작 시 이전 열린 세션 종료" 모두 제거. 회독은 명시적으로만 닫힌다(새 회독 시작 시 그 책의 열린 회독만 종료).
  5. **역할 분리:**
     - `currentRoundFor(bookId)` — 현재 회독 get-or-create. **아무것도 닫지 않음.** 모든 캡처 진입(홈·책장·Review의 📷/✍️)과 공유 수신이 이 경로. 연타해도 회독이 늘지 않는다.
     - `startNewSession(bookId, project?)` — **새 회독 시작 전용.** 그 책의 열린 회독만 닫고(`endOpenRoundsForBook`) 새 회독 생성. 신규 책 등록 직후와 Review의 "새 회독 시작"만 사용.
  6. **홈 = 최근 책 목록**(회독 배지 포함, `recentBooks`/`BookView`). "이어 읽기" 세션 카드 제거. 책장(books)도 같은 패턴: 행 탭=책 Review, 📷/✍️=현재 회독 캡처.
- **ADR-005 개정 메모:** "종료 3경로" 중 (2) 8h 자동 종료와 (3) 타책 시작 시 종료를 폐지. (1) 명시적 종료는 "새 회독 시작"으로 대체. Book↔Session 1:N과 명시적 시작은 유지.
- **함의:**
  - UI 문구 전면 교체: "세션" → "회독"(Export scopeLabel·PDF 파일명, 백업 복원 flash, 책 삭제 confirm, 회독 시작 화면).
  - 코드 식별자(`Session`, `sessionId`, `startNewSession` 등)는 호환성 위해 유지 — 용어 교체는 사용자 노출 문구·문서에 한정.
  - 열린 회독이 여럿인 레거시 데이터: `currentRoundFor`가 가장 최근 것을 선택해 자연 수렴.
- **2026-07-02 개정 — 회독 UI 잠복:** 회독 노출이 불필요하다는 사용자 판단. UI만 제거(배지/새 회독 시작/회독 구분/✎ 번호·제목 편집/이 회독 삭제), 데이터 모델(Session=회독, roundNo)과 db 헬퍼는 보존 — 재도입 시 UI만 복원. 목록 구조는 날짜 구분선으로 대체.

## ADR-017 — 외부 API 예외: 알라딘 TTB로 책 표지 조회 (ADR-006·PRD §19 개정)

- **상태:** 확정 (2026-07-02)
- **맥락:** 수동 등록만으로는 표지가 그라데이션 박스뿐이라 책장·홈에서 책을 시각적으로 구분하기 어렵다. "서버·외부 API 없음"(ADR-006, PRD §19) 원칙은 유지 가치가 크지만, 표지 조회는 *책당 1회*의 저빈도 부가 기능이라 예외 비용이 낮다. 2026-07-02 실측으로 후보를 검증했다:
  - **알라딘 TTB:** JSONP(`callback`) 정상 동작, 표지 이미지 CDN이 `Access-Control-Allow-Origin: *`라 fetch로 바이트 수신 가능. 한국서 커버리지 최상.
  - **대안 전멸:** 카카오 책 검색 = API CORS는 되나 표지 이미지 CDN에 CORS 없음 → 바이트 저장 불가, 해상도도 낮음. 네이버 = API에 CORS 없음(서버 프록시 필요). Google Books = 무키 호출이 사실상 차단(429). OpenLibrary = 한국서 커버리지 사실상 없음.
- **결정:**
  - **알라딘 TTB API를 유일한 외부 API 예외로 허용**(`src/lib/aladin.ts`). 검색은 JSONP, 표지 다운로드는 CDN fetch(CORS `*`).
  - **TTB 키는 사용자 입력** — 설정(백업·설정)에서 입력, `localStorage`(`capture.aladinTtbKey`) 저장. 공개 리포에 키를 커밋하지 않는다. **키가 없어도 앱 전 기능 정상**(표지 찾기 버튼만 미노출).
  - **표지는 1회 수신 후 IndexedDB에 ArrayBuffer로 저장**(`Book.cover`/`coverType`, ADR-015 정합). 이후 표시·백업은 전부 로컬 — 오프라인 동작 유지, 외부 URL 상시 의존 없음.
  - **JSONP 신뢰 트레이드오프 명기:** JSONP는 외부 스크립트 실행이라 알라딘 응답을 신뢰하는 구조다. 콜백 화이트리스트·타임아웃·응답 형태 검증으로 하드닝하되, 근본적으로 알라딘이 악의적이면 방어 불가 — 저빈도·사용자 개시 동작에 한정해 수용한다.
- **ADR-006·PRD §19 개정 메모:** ADR-006의 "MVP 외부 API 없음" 근거와 PRD §19 Out of Scope의 해당 원칙을 **"외부 API 없음(예외: 알라딘 TTB 표지 조회, 사용자 키·1회 수신·로컬 저장)"**으로 개정. ISBN 바코드 스캔은 여전히 Phase 2+.
- **함의:**
  - 표지 렌더는 `cover instanceof ArrayBuffer` 가드 후 `Blob` 재구성 → objectURL `<img>`(ADR-013: `createImageBitmap`/`decode` 금지). objectURL은 화면 재렌더·cleanup에서 revoke.
  - 백업 JSON은 표지를 base64로 직렬화/복원(라운드트립 보존).
  - 네트워크 실패·키 오류는 표지 찾기 UI 안에서만 조용히 처리 — 캡처 루프·핵심 흐름에 영향 없음.
