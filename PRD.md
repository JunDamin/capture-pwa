## 0. 이 문서의 사용법

이 PRD는 기능 명세가 아니라 **의사결정의 기준**이다.
새 기능을 넣을지 말지 고민될 때, 5장(Design Principles)과 16장(Capture Loop Budget)으로 돌아온다.
개별 결정의 *이유*와 트레이드오프는 `docs/decisions.md`(ADR)에 있다.

## 1. Vision

**Mission**
독서 중 떠오르는 생각을 가장 적은 마찰로 기록하고, AI가 나중에 그 생각을 지식으로 성장시킬 수 있도록 한다.

기록의 목적은 *저장*이 아니라, 생각을 **미래의 자신과 AI가 다시 사용할 수 있는 자산**으로 만드는 것이다.

---

## 2. Problem Statement

사람들은 책을 읽으며 수많은 생각을 한다. 하지만 대부분은 기록되지 않는다.

- 기록이 귀찮다.
- 타이핑이 느리다.
- 저장 위치를 고민한다.
- 독서 흐름이 끊긴다.

결국 가장 가치 있는 생각들이 사라진다.

---

## 3. Product Goal

사용자가 *"기록해야지"* 라고 **의식하지 않아도** 자연스럽게 생각을 Capture하도록 만든다.

OCR · 관계 분석 · 인터뷰 · 노트 생성은 모두 **나중에, (외부) AI가** 수행한다.

---

## 4. Design Principles

1. **독서를 방해하지 않는다.**
2. **입력은 최소, 정리는 최대.**
3. **원본보다 생각이 중요하다.** 사진은 입력일 뿐이고(원본 미보관), 최종 자산은 생각·관계·질문·노트다.
4. **AI는 입력 이후에만 등장한다.** 독서 중에는 AI가 개입하지 않는다.
5. **데이터는 항상 이동 가능해야 한다.** 특정 플랫폼에 종속되지 않는다.

---

## 5. 개발 원칙 (North Star)

> **캡처 루프를 느리게 하는 기능은 거부한다.**
> **캡처 *이후*의 기능(Review·Export·AI)은 얼마든지 풍부해도 좋다.**

- 이 원칙은 **사진 경로의 캡처 루프**(사진 → 태그 → Why → 저장 → 카메라 복귀)에만 적용된다.
- "느리게 만드는가?"라는 주관적 토론을 **16장의 Budget을 초과하는가?** 라는 측정으로 환원한다.
- OCR · 음성 · AI · 동기화 등 어떤 기능이든, 캡처 루프 예산을 깨면 MVP에서 추가하지 않는다.

## 6. Target User

**1차** — 종이책을 많이 읽는 사람: PKM/Obsidian 사용자, 연구자, 학생, 개발자, 독서가
**2차** — 전자책 사용자

## 7. User Journey

```
책을 읽는다 → 좋은 문장/생각 발견 → Capture 실행
 — 사진 경로: 찰칵 → 태그 → Why(칩 탭) → 저장 → 바로 독서 복귀
 — 메모 경로: (사진 없이) 메모 입력 → 태그 → 저장
독서 종료(또는 8h 비활동 자동 종료) → Review → Export(prompt.md + 사진 공유)
 → ChatGPT / Claude → 독서노트 생성
```

## 8. Core Features

### A. Reading Session
책을 선택하고 세션을 시작한다(명시적 시작). 상단에는 **항상** 다음을 표시한다.
- 📚 책 제목
- 🎯 프로젝트(선택)
- 📍 Capture 개수

**세션 생명주기 (ADR-005):**
- Book ↔ Session = **1:N** — 한 책을 여러 날에 걸쳐 읽으면 세션이 여러 개.
- **종료(느슨):** ① 사용자가 "세션 종료" ② 마지막 캡처 후 **8시간** 비활동 시 자동 종료 ③ 다른 책으로 새 세션 시작 시 이전 세션 종료.
- **이어읽기:** 열린 세션은 "이어 읽기"로 계속, 닫힌 책은 같은 Book의 새 세션으로 재개.
- **중단 복구:** 매 단계 IndexedDB 즉시 커밋 → 강제 종료 후 재실행 시 세션 복구.

### B. Capture
캡처는 두 경로를 가진다 (ADR-001).
- **사진 경로(빠름):** 카메라 항상 켜짐 → 찰칵. **3초 예산 적용.**
- **메모 경로:** 사진 없이 텍스트 메모로 캡처. 타이핑 허용, 예산 미적용.

**유효성 규칙 (ADR-004):**
- 필수: **(사진 또는 메모 중 최소 하나) + 태그(단일).**
- 선택: **Why · 메모(사진 경로일 때).**
- 타이핑은 모든 경로에서 *항상 가능*하되, 빠른 경로 완료에 *필수는 아니다*(ADR-001 보강).

### C. Review (규칙 기반 분류 + 프롬프트 빌더, AI 호출 없음 — ADR-007)
Capture를 단순 나열하지 **않는다.** AI에게 전달하기 위한 구조를 *규칙으로* 만든다.
- **태그별 묶음/카운트** (⭐ 중요 N, ❓ 의문 N, 🌱 아이디어 N …)
- **Why 칩별 묶음** ("글감" 후보 N, "프로젝트에 써먹기" N) — PRD의 "글감/인터뷰 추천"을 규칙으로 근사.
- **시간순 타임라인** + **"Why 없는 캡처"** 별도 수집(사후 보강 유도).
- 의미 분석(주제·반복·관계)은 **하지 않는다** → Export로 외부 AI에 위임. Review의 결론은 **"Export" 버튼**이다.
- 범위: **"이번 세션" 또는 "이 책 전체"** 선택.

### D. Export (ADR-008/009)
모든 콘텐츠를 **파일로** 내보내 일관된 핸드오프를 만든다.
- **1순위(모바일): Web Share API** — `navigator.share({ files: [prompt.md, 사진들] })`.
- **보조:** "프롬프트 복사"(클립보드), ZIP/이미지 다운로드.
- **`prompt.md`** = 지시문 + 태그 범례 + 구조화된 캡처 데이터 + "첨부 사진을 OCR해 각 캡처와 엮어라" 지시. **버전된 고정 템플릿 1종**(`prompt-template v1`)으로 일관성 확보. (목적별 선택은 Phase 2)

## 9. Tag System (단일 선택 — ADR-002)

태그는 *감정*이 아니라 **AI에게 주는 대표 힌트**다. 한 캡처당 **1개**. 첫 사용 시 의미를 1회 안내한다.

| 태그 | 의미 |
|------|------|
| 💡 | 흥미롭다 |
| ⭐ | 중요하다 |
| 🔗 | 연결된다 |
| ❓ | 의문이다 |
| 🌱 | 아이디어 |

---

## 10. Why — 핵심 차별점

입력창의 이름은 *메모*가 아니라 **"왜 저장했나요?"** 다. "무엇을"이 아니라 "왜"를 물어 의도(intent)를 보존한다.

- **선택 항목이다** (ADR-004). 좋은 문장을 본 순간 "왜"가 늘 떠오르진 않으므로 강제하지 않는다. Review에서 "Why 없는 캡처"를 모아 사후 보강.
- **입력 방식:** **칩(chip) 탭** 기본 + **항상 열려 있는** 선택적 자유입력.
  - `프로젝트에 써먹기` · `다시 읽고 싶음` · `반대되는 생각` · `글감` · `(직접 입력…)`
- 자유입력은 숨겨진 기능이 아니라 늘 한 번에 닿는 곳에 있다(ADR-001 보강).

## 11. Data Model

```
Book (1) → (N) Session (1) → (N) Capture
```

**Book** (ADR-006) — `uuid` · `title`(필수) · `author`(선택) · `isbn`(선택) · `cover`(선택)
**Session** (ADR-005) — `uuid` · `bookId` · `project`(선택) · `started` · `ended`(nullable, 열린 세션=null)
**Capture** (ADR-001/003/004) —
- `uuid` · `sessionId` · `createdAt` · `updatedAt`
- `image: Blob | null` — 리사이즈+압축(긴 변 ~1600px, JPEG ~0.7, 200–500KB). 원본 미보관.
- `memo: string | null`
- `tag: TagEnum` (**NOT NULL**, 단일)
- `why: string | null`
- `ocr: string | null` (선택, MVP 미사용)
- `exportStatus`
- **유효성:** `NOT (image IS NULL AND memo IS NULL)`

---

## 12. Sync Strategy

- 기본 저장: **IndexedDB** (이미지는 Blob, Base64 금지 — ADR-003)
- Export / Import: **ZIP**
- 병합: **UUID 기반 Merge**, **Hash 기반 중복 탐지**
- 서버 없음, 로그인 없음.
- 용량: `navigator.storage.estimate()`로 쿼터 추적, 임계 시 Export/정리 유도.

---

## 13. AI Workflow

```
Capture → Review(규칙 분류) → Export(prompt.md + 사진) → (외부) ChatGPT / Claude
 → 사진 OCR · 주제/반복/관계 분석 · 인터뷰 질문 · 독서노트 생성
```

AI는 앱 내부가 아니라 **Export된 prompt.md + 첨부 사진**을 통해 외부 LLM에서 동작한다 (MVP).

---

## 14. OCR Policy

OCR은 앱의 핵심 기능이 **아니다.** 기본값은 앱 내 OCR 없음. **사진을 외부 LLM에 첨부로 전달해 거기서 OCR**되게 한다(13장). 캡처 루프 안에서는 절대 수행하지 않는다(원칙 5 위배). 앱 내 선택 OCR/AI Vision은 Phase 4.

---

## 15. 캡처 보조 (사진 검색성)

OCR 없이 사진만 찍으면 나중에 무엇을 붙잡았는지 못 찾는다.
→ 사진 촬영 시 **해당 문장 영역을 탭/동그라미로 1회 표시**하는 옵션을 검토 (단, 예산을 깨지 않는 선에서만).

---

## 16. Performance Goal — Capture Loop Budget

> **이 수치는 표어가 아니라 예산이다. 측정되고, 회귀하면 머지하지 않는다.**
> 적용 대상은 **사진 경로**. "타이핑 금지"가 아니라 **"타이핑 없이도 완료 가능"** 으로 읽는다(ADR-001 보강).

| 항목 | 목표 |
|------|------|
| 앱 실행 → 카메라 가동 (웜업) | **≤ 1s** |
| 캡처 1회 완료 (사진→태그→Why→저장) | **p95 ≤ 3s** |
| 저장 → 카메라 복귀 | 즉시 (압축은 백그라운드) |

**최우선 검증 리스크 (프로토타입):**
1. PWA(특히 iOS Safari)에서 카메라 웜업 ≤ 1s 달성 여부.
2. ChatGPT/Claude 앱이 **`.md` + 다중 이미지 혼합 공유**를 한 메시지로 수용하는지(ADR-008).

## 20. Product Statement

Capture는 독서 기록 앱이 아니다.
Capture는 **사람의 경험을 AI가 이해할 수 있는 형태로 구조화하는 입력 인터페이스**다.

기존 독서 앱은 *노트*를 만든다. Capture는 *생각*을 붙잡는다.
기존 PKM은 *정리*를 시작한다. Capture는 *AI가 잘 정리할 수 있도록* 경험을 구조화한다.
